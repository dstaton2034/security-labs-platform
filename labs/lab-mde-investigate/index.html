Investigate and Respond to a Suspicious Process Alert

You'll walk through a real alert in Microsoft Defender for Endpoint — from initial triage to device isolation and remediation — using the exact steps you'd follow on the job.

⏱ ~30 minutes
📊 Foundational
🌐 Real Environment Required
🔐 MDE P2 License
Your Progress
0%

📋 Before You Start

  • You have access to Microsoft Defender XDR portal (security.microsoft.com)
  • At least one device is onboarded to Defender for Endpoint in your tenant
  • Your account has the Security Reader role or higher in Defender
  • You have a test device you can safely isolate (VM recommended)
1

Open the Incidents Queue

Navigate to the alert that needs investigation

security.microsoft.com › Incidents & Alerts › Incidents
Sign into security.microsoft.com with your work account. In the left navigation, expand Incidents & Alerts and click Incidents.
1
In the search/filter bar, set Status = Active and Severity = High or Medium
2
Find an incident that contains a Suspicious Process or Malware alert — click its title to open it
3
Note the affected device name and alert timestamp — you'll need these in later steps
✓ Expected Result

You should see the Incident page with an Attack story graph, a list of related alerts, and the affected entities (device + user) listed on the right panel.

If your environment has no active incidents, go to Alerts instead (same menu) and filter by Detection source: Microsoft Defender for Endpoint. You can also generate a test alert by running the EICAR test file on an onboarded device.
2

Review the Alert Details and Attack Timeline

Understand what happened, when, and on which device

Incident › Attack story
Inside the incident, click the Attack story tab. This shows the full process tree — what ran, what it spawned, and what it touched.
1
Click the root process (top of the tree) — check the file name, path, and parent process
2
Look for red nodes (malicious activity) vs yellow nodes (suspicious) in the tree
3
Click any red node → read the Alert description and MITRE ATT&CK technique shown in the right panel
4
In the Evidence tab, confirm whether files, registry keys, or network connections were involved
✓ Expected Result

You can identify: which process triggered the alert, its parent, the MITRE technique (e.g. T1059 Command and Scripting Interpreter), and whether any files were dropped or network connections made.

The process tree reads top-down — parent processes are at the top, child processes branch below. A suspicious PowerShell process spawned by winword.exe (Word) is a classic sign of a macro-based attack (T1566 Phishing).
3

Run a KQL Hunt for Related Activity

Use Advanced Hunting to see if other devices were affected

security.microsoft.com › Hunting › Advanced Hunting
Navigate to Hunting → Advanced Hunting in the left nav. Run the query below — replace DeviceName with the device from Step 1. This looks for similar process behavior across your environment in the last 7 days.
KQL — Advanced Hunting
// Hunt for suspicious PowerShell spawned by Office apps DeviceProcessEvents | where Timestamp > ago(7d) | where InitiatingProcessFileName in~ ( "winword.exe", "excel.exe", "powerpnt.exe", "outlook.exe") | where FileName in~ ("powershell.exe", "cmd.exe", "wscript.exe") | project Timestamp, DeviceName, InitiatingProcessFileName, FileName, ProcessCommandLine | order by Timestamp desc
1
Paste the query → click Run query (Shift + Enter)
2
If results appear — check the DeviceName column. Multiple devices = wider spread, escalate immediately
3
Click Create detection rule (top right) to turn this into a recurring scheduled alert
✓ Expected Result

Query returns results showing which devices, when, and what command line was executed. If only your test device shows up — blast radius is contained. Multiple devices = active spread.

That's actually good news — it means the activity is isolated to one device. Try widening the time window: change ago(7d) to ago(30d). You can also search for the specific filename from the alert in Step 2 using: | where FileName =~ "suspicious_file.exe"
4

Isolate the Device

Cut off the affected device from the network while keeping Defender communication open

Incident › Affected assets › Device page › Actions
Go back to the incident. Under Affected assets → Devices, click the device name. On the device page, click the … Actions menu (top right) and select Isolate device.
1
Click Isolate device → in the dialog, type a comment: "Isolating due to suspicious process alert — [your name]"
2
Select isolation type: Full isolation (blocks all network traffic except Defender telemetry)
3
Click Confirm — the action is queued and will execute within seconds if device is online
4
Check the Action center (left nav → Action center) to confirm the isolation was successful
✓ Expected Result

Device status changes to "Isolated" on its page. Action center shows Status: Succeeded. The device is now cut off from the network but still sending telemetry to Defender — you can still run live response on it.

Pending means the device hasn't checked in yet. This happens if the device is offline or asleep. The action will execute the next time it connects. You can leave it pending — Defender will isolate it automatically when it comes back online.
5

Run Antivirus Scan and Collect Investigation Package

Remediate the threat and capture forensic evidence

Device page › Actions
Still on the device page, use the Actions menu to run two actions in sequence.
1
Actions → Run antivirus scan → select Full scan → confirm. This triggers Microsoft Defender Antivirus on the device to scan and quarantine threats
2
Actions → Collect investigation package → confirm. This collects running processes, network connections, scheduled tasks, and prefetch files as a downloadable zip
3
Monitor both actions in Action center — the investigation package takes 5–10 minutes to collect
4
Once collected, download the package and examine processes.json and network_connections.json for IOCs
✓ Expected Result

Action center shows both actions as Succeeded. The investigation package is available for download. Any quarantined items appear in the Threat & vulnerability management → Remediation view.

6

Classify the Incident and Release the Device

Close the loop — document your findings and restore the device

Incident › Manage incident
Go back to the Incident page. Click Manage incident (top right panel) to classify and close it properly.
1
Set Classification → True positive (if actual threat) or False positive (if benign)
2
Set Determination → e.g. Malware, Security testing, or Unwanted software
3
Add an Incident comment summarizing what you found and what actions you took
4
Set Status → Resolved → click Save
5
Return to the Device page → Actions → Release from isolation once you've confirmed the threat is remediated
✓ Expected Result

Incident status changes to Resolved. Device is released from isolation and shows Active status. Your classification feeds into Microsoft's threat intelligence to improve future detections.

Microsoft uses your true/false positive classifications to tune detection models across all tenants. Accurate classification = better alerts for the whole ecosystem. It also creates a paper trail for compliance audits.
