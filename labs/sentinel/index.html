🧠 Deep Dive

How Data Ingestion & Log Analytics Actually Work

This is the engine room of Sentinel. Understanding how data gets from a firewall or a laptop into a queryable table is the foundation of everything else. Let's break it down piece by piece.

📦 What IS a Log Analytics Workspace?

Think of it as a giant cloud database that stores all your security logs. It's where EVERY piece of data in Sentinel lives — sign-in logs, firewall logs, endpoint alerts, cloud activity, everything.

🗄️ It's Organized into Tables

Each data source gets its own table. SigninLogs for Entra ID logins. SecurityEvent for Windows events. AzureActivity for Azure resource changes. CommonSecurityLog for CEF/syslog from firewalls. You query these tables with KQL.

💰 You Pay for What Goes IN

Billing is based on data ingestion volume (GB/day). More logs = higher cost. That's why filtering and choosing the right log tier matters. You can also commit to a daily volume for discounts (commitment tiers).

⏱️ Retention = How Long Data is Kept

Interactive retention (default 90 days): data you can query instantly with KQL. Long-term retention (up to 12 years): archived data you can retrieve with a search job when needed. Compliance often requires 1-7 years.

🔒 It's Tamper-Proof

Log Analytics is an append-only platform. Once data is written, it can't be modified or deleted (except by policy for compliance). This means attackers can't cover their tracks by deleting logs.

🔄 The Ingestion Pipeline — How Data Gets From Source → Table

1

🏢 Data Sources (Where Logs Come From)

Your environment generates logs everywhere: Entra ID sign-ins, Azure resource activity, Windows/Linux servers, firewalls (Palo Alto, Fortinet, Cisco), cloud apps (M365, AWS), custom apps via API. Each source produces logs in its own format.

2

🔌 Data Connectors (The Pipes)

Connectors are the "pipes" that move data from source to workspace. 300+ built-in connectors. There are different types depending on the source:

🔷 Microsoft Native (One-Click)

Entra ID, M365, Defender XDR, Azure Activity — just toggle ON. Data flows in seconds. No agents, no forwarders, no config.

How it works →
  • Uses Azure diagnostic settings or built-in service integration
  • Data goes directly from the Microsoft service → your workspace
  • Examples: SigninLogs, AuditLogs, SecurityAlert, AzureActivity
  • Typically free or included in your M365/Azure license

🔥 Syslog / CEF (Firewalls & Network)

Palo Alto, Fortinet, Cisco ASA, Check Point — send logs via Syslog/CEF to a Linux forwarder running Azure Monitor Agent.

How it works →
  • Firewall sends syslog to a Linux VM (the "forwarder")
  • Azure Monitor Agent (AMA) on the forwarder collects and forwards to workspace
  • CEF = Common Event Format — a standardized syslog format for security
  • Data lands in the CommonSecurityLog (CEF) or Syslog table
  • You need a DCR (Data Collection Rule) to specify what to collect

🖥️ Agent-Based (Servers & VMs)

Azure Monitor Agent (AMA) installed on Windows/Linux servers. Collects Windows Event Logs, performance counters, custom logs.

How it works →
  • AMA installed on the VM (Azure-native, Arc for on-prem/other clouds)
  • Data Collection Rules (DCRs) define exactly which logs to collect
  • Windows: Security events, Application logs, custom event channels
  • Linux: syslog facilities (auth, daemon, kern), custom log files
  • Data lands in SecurityEvent, Event, or custom tables

🔗 REST API / Custom (Anything Else)

For sources without a built-in connector — use the Logs Ingestion API to push data from any application via HTTP POST.

How it works →
  • Your app sends JSON data via HTTPS POST to the ingestion endpoint
  • A DCR defines the schema and any transformations
  • Data lands in a custom table (e.g., MyApp_CL)
  • Can also use Logic Apps or Azure Functions as middleware
  • Codeless Connector Platform for building connectors without code
3

🔄 Data Collection Rules & Transformation

Before data hits the workspace, Data Collection Rules (DCRs) can filter, transform, and enrich it. Think of DCRs as the rules that decide what gets in and what gets thrown away. This is where you save money — filter out noise before you pay to store it.

4

📊 Log Analytics Tables (Where Data Lives)

Data lands in tables organized by source. Each table has a plan that controls cost and query capability. This is where you choose how much to spend vs. how much you can do with the data.

💰 Table Plans — The Cost vs. Capability Tradeoff

Analytics Logs

📊

Full KQL query capability. Alerts, visualizations, Fusion, analytics rules — everything works. Default 90-day interactive retention.

Full Power Standard Cost

Use for: Security events, sign-in logs, alerts — data you query constantly and build detections on.

Basic Logs

📋

Limited KQL — can query but some operators restricted. Can't use for analytics rules or alerts. 30-day interactive retention.

~60% Cheaper Query Only

Use for: Verbose logs you keep for investigations but don't alert on — debug logs, verbose firewall flows, trace data.

Auxiliary Logs

🗄️

Minimal query — primarily for long-term compliance storage. Very limited KQL. 30-day interactive, up to 12-year long-term retention.

~85% Cheaper Archive

Use for: Compliance data you must keep for years but rarely query — audit trails, access logs, regulatory records.

📋 Key Tables You'll Work With

SigninLogs
Entra ID interactive sign-ins — who logged in, from where, MFA status, risk level, device
SecurityEvent
Windows Security Event Log — logons, process creation, privilege use (from AMA agent)
SecurityAlert
Alerts from Defender XDR, MDC, and other security products — the alerts themselves
AzureActivity
Azure control plane operations — who created/deleted/modified what resource
CommonSecurityLog
CEF-formatted logs from firewalls (Palo Alto, Fortinet, Cisco) — network security events
Syslog
Linux syslog messages — auth, daemon, kernel events from Linux servers
SecurityIncident
Sentinel incidents — the correlated groupings of alerts that analysts investigate
DeviceProcessEvents
MDE endpoint process telemetry (via XDR connector) — command lines, parent processes, hashes

🔍 Real-World Example: A Firewall Log's Journey

Here's exactly what happens when your Palo Alto firewall generates a "connection blocked" event:

1. Palo Alto generates a CEF syslog message: "CEF:0|PAN|Firewall|10.0|THREAT|url|3|src=10.0.1.50 dst=185.220.101.42 act=blocked"
2. Syslog forward → message sent over TCP/514 to your Linux forwarder VM
3. Azure Monitor Agent on the forwarder picks it up (configured via DCR to listen on the CEF facility)
4. DCR transformation → parses the CEF fields, extracts src IP, dst IP, action, severity
5. Ingested → lands in the CommonSecurityLog table in your workspace
6. Queryable → within seconds, you can query it: CommonSecurityLog | where DestinationIP == "185.220.101.42"
7. TI Matching → Sentinel auto-checks destination IP against threat intelligence → if it's a known C2 IP, alert fires automatically
💡 Why Understanding Ingestion Matters
💰

Cost Control

Data ingestion is the #1 Sentinel cost driver. If you don't understand table plans, DCR filtering, and commitment tiers, you'll overpay massively. One verbose firewall can cost more than your entire SIEM budget.

🔗

No Data = No Detection

If a data source isn't connected, Sentinel is blind to it. Understanding connectors means you know exactly what Sentinel can and can't see. Coverage gaps = security gaps.

📊

Table Structure = Query Capability

If you know the tables, you can write KQL. If you can write KQL, you can hunt, detect, and investigate. Every analytics rule, every hunt, every workbook starts with knowing which table to query.

⚡

DCR Transforms Save Money & Improve Signal

Filter out 80% of verbose Windows security events at ingestion time (keep logon events, drop everything else). You pay less AND your detections have better signal-to-noise ratio.

🎯 Pillar 2

Threat Detection — Analytics Rules

Four types of analytics rules detect threats from known patterns to unknown multi-stage attacks. All map to MITRE ATT&CK for coverage visibility.

📅

Scheduled Rules

KQL queries that run on a schedule (every 5 min to every 14 days). The workhorse of detection — you write the query, set the threshold, map to MITRE.

Custom KQL
Details
  • Fully customizable KQL queries
  • Entity mapping (account, IP, host, file)
  • Alert grouping into incidents
  • Event grouping (all events or first event only)
  • Hundreds of built-in templates from Content Hub
⚡

Near-Real-Time (NRT)

Run every 1 minute with a 1-minute lookback. For time-critical detections where 5-minute delay is too long — ransomware, active exploitation.

~1 min latency
Details
  • 1-minute run frequency
  • Subset of KQL supported (optimized for speed)
  • Ideal for high-severity, time-critical threats
  • Same entity mapping and incident creation as scheduled
🧬

Fusion (ML-Powered)

Microsoft's ML engine automatically correlates low-fidelity alerts from multiple sources into high-confidence multi-stage attack detections. Catches what individual rules can't.

AI/ML
Details
  • Correlates alerts from Defender XDR, MDC, Entra ID, and custom rules
  • Detects multi-stage attacks (e.g., phishing → credential theft → data exfil)
  • 150+ attack pattern detections built-in
  • No tuning needed — Microsoft manages the ML models
📊

Anomaly Rules

Behavioral analytics baseline what's "normal" for your environment, then flag deviations. Catches insider threats and novel attacks without pre-defined signatures.

Behavioral
Details
  • Machine learning baselines per-entity behavior
  • Anomalous login locations, times, volumes
  • Unusual data access or download patterns
  • Configurable thresholds and scoring
  • Results appear as anomalies that enrich incidents
💡 Why Layered Detection Matters
🧬

Fusion Catches Multi-Stage Attacks

A single failed login is noise. But failed login (Entra ID) + new inbox rule (M365) + unusual file download (SharePoint) + MDE alert = Fusion connects the dots into one high-confidence incident.

⚡

NRT for Time-Critical Threats

When ransomware encrypts files, every minute counts. NRT rules detect and trigger response within 1 minute — vs 5-15 minutes for scheduled rules.

📊

Anomalies Find What Rules Can't

You can't write a rule for "something weird." Anomaly detection baselines normal behavior and flags deviations — catches insider threats and novel TTPs without signatures.

🗺️

MITRE Coverage Map

Sentinel shows which ATT&CK techniques you have detections for and which are gaps. This turns "are we protected?" from a gut feeling into a data-driven answer.

When an incident fires, Sentinel provides deep investigation tools — interactive entity graphs, threat hunting with KQL, watchlists, bookmarks, and Jupyter notebooks.

🕸️

Incident Investigation Graph

Interactive graph showing entities (users, IPs, hosts, files) and their connections. Click an entity to expand its timeline and related alerts.

Details
  • Visual entity relationship mapping
  • Drill down into any entity's full timeline
  • See which alerts are connected and why
  • Expand to discover related entities not in the original alert
🎯

Threat Hunting

Proactive KQL-based hunting across all ingested data. MITRE-organized built-in queries. Save results as bookmarks, promote to incidents.

Details
  • Hundreds of built-in hunting queries organized by MITRE tactic
  • Livestream — run a query continuously and get alerted on new results
  • Bookmarks — save interesting findings for later investigation
  • Hunts — collaborative campaigns with hypothesis tracking
📋

Watchlists

Upload reference data (VIP users, terminated employees, allowed IPs) and correlate with security events in KQL queries and analytics rules.

Details
  • CSV upload or Azure Storage integration
  • Use in KQL: _GetWatchlist('VIPUsers') for enrichment
  • Use in analytics rules to reduce false positives or increase severity
  • Auto-refresh from external sources
📓

Jupyter Notebooks

Azure ML-integrated notebooks for advanced analysis — Python ML libraries, custom visualizations, data from sources beyond Sentinel.

Details
  • Full Python data science stack (pandas, scikit-learn, matplotlib)
  • MSTICPy library for security-specific analysis
  • Query Sentinel data + external sources in one notebook
  • Custom process trees, network graphs, timeline visualizations
💡 Why Investigation Tools Matter
🕸️

Graphs Beat Log Scrolling

Staring at raw logs to find the story is slow and error-prone. The investigation graph shows relationships visually — "this IP connected to this user who accessed this file" — in seconds.

🎯

Hunting = Proactive Security

Waiting for alerts is reactive. Hunting lets your senior analysts look for threats before they trigger detections — reducing dwell time from months to days.

📋

Watchlists Close Context Gaps

"Is this user a VIP?" "Is this IP on our allowed list?" Without watchlists, analysts waste time looking up context. With them, the context is embedded in every query.

📓

Notebooks for Advanced Analysis

When KQL isn't enough — ML anomaly detection, clustering analysis, custom visualizations — notebooks give data-science-level capability without leaving the Sentinel ecosystem.

⚡ Pillar 4

SOAR — Automation & Response

Two automation layers: Automation Rules for lightweight incident triage, and Playbooks (Logic Apps) for complex orchestration across any service.

⚙️

Automation Rules

Lightweight if/then rules that run on every incident. Auto-assign, auto-tag, change severity, close false positives, or trigger playbooks — no code required.

Details
  • Run automatically on incident creation or update
  • Conditions: title contains, severity equals, entity matches, etc.
  • Actions: assign owner, add tags, change status/severity, run playbook, suppress
  • Order of execution is configurable (priority-based)
  • Built-in expiration dates for temporary rules
🔄

Playbooks (Logic Apps)

Full orchestration workflows built on Azure Logic Apps. 200+ connectors — ServiceNow, Slack, Teams, Jira, PagerDuty, custom APIs. Trigger on incidents or alerts.

Details
  • Visual workflow designer — no code required
  • 200+ connectors: Teams, Slack, ServiceNow, Jira, PagerDuty, custom REST APIs
  • Enrich incidents: lookup IP reputation, check threat intel, query user details
  • Response actions: block IP in firewall, disable user in Entra ID, isolate device via MDE API
  • Notify: post to Teams/Slack channel, send email, page on-call via PagerDuty
  • Ticket: auto-create ServiceNow/Jira tickets with full incident context
💡 Why SOAR Matters
⏱️

MTTR: Hours → Seconds

A playbook that disables a compromised user, isolates their device, and posts to the SOC channel runs in 10 seconds. A human doing the same across 3 portals takes 20 minutes.

📉

Triage at Machine Speed

Automation rules close known false positives, auto-assign by source, and escalate critical incidents — before an analyst even sees them. Your tier 1 workload drops dramatically.

🔗

200+ Service Connectors

Your SOC doesn't live in one tool. Playbooks bridge Sentinel to your firewall, ticketing system, chat platform, identity provider, and custom APIs — unified response across your stack.

📊

Consistent, Auditable Response

Humans make different decisions under pressure. Playbooks follow the same steps every time. Every action is logged — perfect for compliance audits and post-incident reviews.
