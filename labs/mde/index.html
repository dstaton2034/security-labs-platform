⚡ Layer 1

Endpoint Sensor Layer

The MsSense.exe sensor runs on each endpoint, hooking into the OS at the kernel and user-mode levels to collect behavioral telemetry. On Linux, it uses eBPF instead of kernel modules for stability.

🔴 Simulated Telemetry Stream
📡

ETW / eBPF — Process Events

Hooks into kernel-level process creation callbacks and ETW providers. On Linux, uses eBPF tracepoints attached to syscalls.

DeviceProcessEvents
Click for details
  • Full command-line arguments for every process
  • Parent-child process relationships (process tree)
  • SHA1 / SHA256 file hashes at creation
  • Account name, SID, and token elevation status
  • Timestamp with millisecond precision
  • Internal throttling prevents duplicate event flooding
📁

Minifilter Driver — File Events

A file system minifilter driver intercepts all file I/O operations. On Linux, fanotify and eBPF provide equivalent coverage.

DeviceFileEvents
Click for details
  • File creates, modifications, deletes, renames
  • Full file path, size, and hash
  • Which process performed the operation
  • Detects file drops in temp/startup folders
  • Tracks ransomware-like bulk encryption patterns
🌐

WFP / Socket Hooks — Network Events

Windows Filtering Platform callouts capture all network connections. Full visibility into outbound/inbound traffic without packet capture overhead.

DeviceNetworkEvents
Click for details
  • Remote IP address, port, protocol (TCP/UDP)
  • Remote URL and DNS query resolution
  • Which process initiated the connection
  • Bytes sent/received
  • Connection direction (inbound/outbound)
  • SmartScreen reputation checked for URLs
🗝️

Registry Callbacks — Registry Events

CmRegisterCallbackEx kernel callbacks monitor all registry key and value operations — critical for detecting persistence mechanisms.

DeviceRegistryEvents
Click for details
  • Key/value creates, modifications, deletes
  • Run/RunOnce keys (persistence)
  • Services registry modifications
  • COM object hijacking detection
  • Full registry path, value name, and value data
🔍

AMSI — Script Content Capture

Antimalware Scan Interface captures deobfuscated script content from PowerShell, VBScript, JScript, and Office VBA — sees the actual code after decoding.

DeviceEvents
Click for details
  • PowerShell script block logging (full content)
  • VBScript and JScript runtime inspection
  • Office VBA macro content analysis
  • Bypasses Base64/encoding obfuscation
  • Paired with cloud ML models for classification
👤

LSA / Security Events — Logon Events

Hooks into the Local Security Authority to capture all authentication events including interactive, RDP, network, and service logons.

DeviceLogonEvents
Click for details
  • Logon type (interactive, RDP, network, service, batch)
  • Success and failure events with reason codes
  • Account name, domain, SID
  • Remote IP for network logons
  • Detects brute force and credential spraying patterns
📦

Image Load Notifications — DLL Events

Kernel-level image load notifications track every DLL loaded into any process — essential for detecting DLL sideloading and injection.

DeviceImageLoadEvents
Click for details
  • DLL file path and hash
  • Digital signature and signer name
  • Process that loaded the DLL
  • Detects unsigned DLLs in signed process spaces
  • DLL search order hijacking detection
🧠

Kernel & Memory Manager — Deep Optics

Deep instrumentation into the kernel memory manager detects process injection techniques like VirtualAlloc, WriteProcessMemory, CreateRemoteThread chains.

DeviceEvents
Click for details
  • Process hollowing detection
  • Remote thread creation monitoring
  • Memory allocation pattern analysis
  • Shellcode injection detection
  • Credential dumping from LSASS memory
💡 Why the Sensor Layer Matters
🎯

Visibility You Can't Get Elsewhere

Without kernel-level hooks, you're blind to process injection, fileless attacks, and in-memory malware. Traditional antivirus only sees files on disk — MDE sees behavior.

⚡

Real-Time, Not After-the-Fact

The sensor streams telemetry as events happen — not from periodic log collection. This is what enables sub-minute detection of attacks in progress.

🔗

Complete Process Trees

Every process has a parent. The sensor captures the full chain — so when PowerShell spawns from Word, you see the entire kill chain, not just an isolated alert.

📊

Low Overhead (~5-20 MB/day)

Enterprise-grade telemetry with minimal network and performance impact. The sensor deduplicates and throttles to prevent flooding — you get signal, not noise.

🛡️ Prevention Pillar 1

Microsoft Defender Antivirus (MDAV)

MDAV is the next-generation antivirus engine built into Windows. It's NOT just signature-based AV — it uses local ML models, cloud-delivered protection, behavior monitoring, and emulation to catch threats traditional AV misses. It's the first line of defense before EDR even kicks in.

⚙️ How MDAV Works — The Protection Stack

📡
Real-Time Protection

Always-on scanning using file and process behavior monitoring. Intercepts files at creation, execution, and modification.

Always On
How it works →
  • On-access scanning: Every file opened/created is scanned before the process can use it
  • Behavior monitoring: Watches API calls, memory operations, and file system activity for malicious patterns
  • Process monitoring: Tracks process creation chains — Word→PowerShell triggers immediate analysis
  • If real-time protection is disabled, MDE generates an alert — tamper protection prevents unauthorized changes
  • Performance tip: Use exclusions for known-safe high-I/O paths (e.g., database files) to reduce scan overhead
☁️
Cloud-Delivered Protection

When a file is unknown locally, metadata (or the file itself) is sent to Microsoft's cloud for deep ML analysis. Verdict returns in milliseconds.

Near Instant
How it works →
  • "Block at first sight": Unknown files are HELD from executing while the cloud analyzes them
  • Cloud uses deep neural networks, detonation sandboxes, and reputation data from 78T daily signals
  • Verdict (allow/block) returns within seconds — user experiences minimal delay
  • Cloud protection level: Default, High, High+, Zero Tolerance — higher levels increase cloud analysis aggressiveness
  • Extended timeout: Configure up to 60 seconds for cloud analysis of suspicious files
  • Requires connectivity: If the device can't reach the cloud, it falls back to local engines only
🔄
Security Intelligence Updates

Signature and ML model updates delivered multiple times per day. Emergency dynamic updates for critical threats in minutes.

Auto-Update
How it works →
  • Security intelligence: Traditional signatures + ML model updates — delivered 3-4 times per day
  • Platform updates: Engine and component updates — monthly via Windows Update
  • Emergency updates: Cloud-delivered dynamic intelligence for critical threats — deployed in minutes globally
  • Update sources: Windows Update, WSUS, SCCM/Intune, or direct from Microsoft Update
  • Outdated sigs = risk: Defender for Cloud generates an alert if signatures are more than 7 days old
🐾
PUA Protection

Potentially Unwanted Applications — adware, bundleware, browser toolbars, crypto-miners. Not malware, but unwanted.

Optional
How it works →
  • Modes: Block (quarantine), Audit (log only), Disabled
  • Detects: adware, bundlers, crypto-miners, browser modifiers, torrent software
  • Configure via: Intune, Group Policy, PowerShell (Set-MpPreference -PUAProtection Enabled)
  • Important for compliance — many frameworks require PUA blocking

🔀 MDAV Operating Modes

✅

Active Mode

Primary AV. Real-time protection ON. Scans and remediates threats. Full protection. This is the default and recommended mode.

⏸️

Passive Mode

Third-party AV is primary. MDAV still scans but does NOT remediate. EDR detection still works. Required for AIR to function. Triggered automatically when another AV is detected.

⛔

Disabled

AV completely off. NO scanning, NO EDR behavioral monitoring, NO AIR. Tamper protection prevents this. If MDAV is disabled, Automated Investigation and Response will NOT function.

⚠️ Key Configuration Gotchas

• Tamper protection: Prevents unauthorized changes to AV settings. Enable it — attackers' first move is often disabling AV
• Exclusions are dangerous: Every exclusion is a blind spot. Attackers actively look for excluded paths to hide malware. Never exclude entire drives or common folders like C:\Windows\Temp
• Cloud protection MUST be on: Without it, you lose "block at first sight" and cloud ML — your protection drops significantly against zero-day threats
• Passive mode + EDR: If you run CrowdStrike as primary AV, MDAV goes passive. EDR (MDE sensor) still works but AIR requires MDAV in active or passive — NOT disabled
• Sample submission: Enable automatic sample submission — this is how Microsoft trains ML models. Without it, your protection is weaker
• Scan type: Quick scan checks common malware locations (~1-2 min). Full scan checks everything (~1-4 hours). Schedule full scans weekly during off-hours
💡 Why MDAV Matters (It's Not Just "Antivirus")
🧬

8 Detection Engines, Not Just Signatures

Traditional AV = signatures. MDAV = signatures + local ML + cloud ML + heuristics + emulation + behavior monitoring + AMSI + memory scanning. It catches polymorphic and zero-day malware that signature-only AV misses completely.

⚡

Built Into Windows = Zero Deployment

No agent to install, no compatibility issues, no performance overhead from a third-party hooking into the kernel. It's already there on every Windows machine. One toggle in Intune enables it enterprise-wide.

🔗

Feeds the Entire MDE Stack

MDAV detections feed into EDR alerts, AIR investigations, Advanced Hunting tables, and the Intelligent Security Graph. Replace it with CrowdStrike AV and you lose this native data pipeline.

🛡️

EDR in Block Mode

Even in passive mode (third-party AV is primary), MDAV can block threats detected by EDR. Called "EDR in block mode" — the EDR detection triggers MDAV to remediate even when it's not the primary AV. Best of both worlds.

🧱 Prevention Pillar 2

Attack Surface Reduction (ASR)

ASR is a collection of hardening rules and controls that block specific attack techniques BEFORE they succeed. While MDAV catches malware, ASR prevents the attack behaviors that LEAD to malware — like Office spawning PowerShell, or credential theft from LSASS.

📋 ASR Rules — The Key Rules You Must Know

Block Office apps from creating child processes

Prevents Word/Excel/PowerPoint from spawning cmd.exe, powershell.exe, etc. This kills 90% of macro-based attacks.

Critical
Details →
  • GUID: d4f940ab-401b-4efc-aadc-ad5f3c50688a
  • Blocks: Word→powershell.exe, Excel→cmd.exe, PowerPoint→mshta.exe
  • Does NOT block: Office opening other Office apps, or legitimate add-ins
  • False positive risk: Low — legitimate Office workflows rarely spawn shell processes
  • Deploy in Audit mode first for 2 weeks, check for false positives, then switch to Block

Block credential stealing from LSASS

Prevents processes from accessing LSASS memory to dump credentials. Stops Mimikatz, procdump, comsvcs.dll MiniDump.

Critical
Details →
  • GUID: 9e6c4e1f-7d60-472f-ba1a-a39ef669e4b2
  • Blocks: Any process opening LSASS with read-memory permissions
  • Works alongside Credential Guard (which uses virtualization to protect LSASS)
  • False positive risk: Medium — some legitimate security tools access LSASS (antivirus, DLP agents)
  • If a legitimate tool is blocked, add a per-process exclusion — NOT a global disable

Block executable content from email

Prevents email clients and webmail from running downloaded executables, scripts, and DLLs.

High Value
Details →
  • GUID: be9ba2d9-53ea-4cdc-84e5-9b1eeee46550
  • Blocks: .exe, .dll, .scr files dropped by Outlook, Thunderbird, webmail
  • Does NOT block: .docx, .pdf, .xlsx (those are handled by other rules)
  • Works with Outlook desktop — webmail coverage depends on browser integration

Block process creations from PSExec and WMI

Prevents PSExec and WMI from creating new processes — key lateral movement tools.

Lateral Movement
Details →
  • GUID: d1e49aac-8f56-4280-b9ba-993a6d77406c
  • Blocks: PsExec.exe and WMI remote process creation (wmiprvse.exe → child process)
  • False positive risk: HIGH — many management tools use WMI and PSExec legitimately (SCCM, monitoring)
  • Must run in Audit mode extensively before enabling Block — check with IT ops team first
  • Exclusions may be needed for specific management servers

Block JavaScript or VBScript from launching downloaded content

Prevents scripts from executing payloads they downloaded from the internet.

Download Cradles
Details →
  • GUID: d3e037e1-3eb8-44c8-a917-57927947596d
  • Blocks the "download cradle" pattern: script downloads .exe → script executes .exe
  • Commonly used in phishing: user runs .js file from email → script downloads malware → script runs it
  • False positive risk: Low — legitimate scripts rarely download and execute binaries

Use advanced protection against ransomware

Uses cloud-delivered protection to block files that look like ransomware based on reputation and ML analysis.

Ransomware
Details →
  • GUID: c1db55ab-c21a-4637-bb3f-a12568109d35
  • Cloud ML evaluates files for ransomware characteristics before allowing execution
  • Requires cloud-delivered protection to be enabled (depends on cloud connectivity)
  • Works alongside Controlled Folder Access for defense-in-depth against ransomware

🧱 Other ASR Components (Beyond Rules)

📁
Controlled Folder Access

Whitelists which processes can write to protected folders. Blocks ransomware from encrypting Documents, Desktop, etc.

Configuration →
  • Default protected folders: Documents, Pictures, Videos, Desktop, Favorites
  • Add custom folders: Set-MpPreference -ControlledFolderAccessProtectedFolders "C:\Finance"
  • Allow specific apps: Set-MpPreference -ControlledFolderAccessAllowedApplications "C:\Apps\MyApp.exe"
  • Gotcha: Many legitimate apps get blocked — LOB apps, backup software, development tools. Test in Audit mode first
  • Gotcha: Only works when MDAV is in Active mode (not passive)
🛡️
Exploit Protection

Per-process mitigations that make memory corruption exploits fail: DEP, ASLR, CFG, heap randomization.

Configuration →
  • System-level: DEP (always on), SEHOP, ASLR (force randomization), CFG (Control Flow Guard)
  • Per-process: Override mitigations for specific apps (e.g., enable strict ASLR for Chrome)
  • Export/import settings as XML for deployment via Intune or GPO
  • Gotcha: Some older apps crash with forced ASLR — test per-app overrides before deploying
  • Replaces the old EMET (Enhanced Mitigation Experience Toolkit)
🌐
Network Protection

Blocks outbound connections to malicious/phishing domains and IPs. Uses SmartScreen reputation data. Works at the network layer — catches all apps, not just browsers.

Configuration →
  • Modes: Block, Audit, Disabled. Set via: Set-MpPreference -EnableNetworkProtection Enabled
  • Uses SmartScreen reputation from the Intelligent Security Graph (78T signals)
  • Key difference from SmartScreen: SmartScreen protects browsers only. Network Protection protects ALL processes
  • Blocks: known C2 servers, phishing domains, malware distribution sites
  • Gotcha: Can block legitimate sites if reputation is wrong — check Network Protection events in portal
  • Web content filtering (URL categories) is also built into Network Protection
🔌
Device Control

Controls USB/removable media — block, allow read-only, or audit by device class, vendor ID, or serial number.

Configuration →
  • Policies via Intune: Device Installation → Device Restrictions → Removable Storage
  • Granular: allow specific approved USB drives by hardware ID, block everything else
  • Options: block entirely, allow read-only, allow read-write for specific devices, audit only
  • Gotcha: Blocking USB can break keyboards, mice, and headsets if rules are too broad — use device class filtering
  • MDE generates alerts in DeviceEvents for removable media activity

🚀 ASR Deployment Best Practice

Phase 1 — Audit (2-4 weeks): Enable ALL ASR rules in Audit mode. Monitor Reports → Attack Surface Reduction in the Defender portal. Identify false positives.
Phase 2 — Warn (1-2 weeks): Switch high-confidence rules to Warn mode. Users see a notification but can bypass. Validates that blocks won't break workflows.
Phase 3 — Block: Enable Block mode for rules with zero/low false positives. Add per-file/per-process exclusions for legitimate business apps that trigger rules.
Phase 4 — Monitor: Continuously monitor ASR events. New apps and workflows may trigger rules. Review monthly.
⚠️ NEVER go straight to Block mode without Audit first. You WILL break business applications. The #1 reason ASR deployments fail is skipping the Audit phase.
💡 Why ASR Is the Most Underused MDE Feature
🛡️

Prevention > Detection

EDR detects attacks. ASR PREVENTS them. Blocking Office from spawning PowerShell stops the attack before malware ever executes. The best alert is the one that never fires because the attack was blocked.

📊

Free With Every MDE License

ASR rules are included in both MDE Plan 1 and Plan 2. Most organizations only use 2-3 rules. There are 16+ rules available — each one closes an attack vector. Turn them on.

🎯

Specifically Targets Real Attack Techniques

Each ASR rule maps to a specific real-world attack technique. "Block Office child processes" kills 90% of macro attacks. "Block LSASS credential stealing" stops Mimikatz. These aren't theoretical — they stop the attacks you see in the news.

⚠️

The Reason People Don't Use It: Fear

Security teams worry ASR will break things. That's what Audit mode is for. Run Audit for 2-4 weeks, fix exclusions, then Block. Every org that does this wishes they'd done it sooner. The risk of NOT using ASR is far greater than the risk of false positives.

Microsoft Defender Antivirus uses a layered defense with paired client-side and cloud-side engines. Each layer catches what the previous might miss. Source: Advanced technologies at the core of Microsoft Defender Antivirus

💻 On the Client

🧠

Memory Scanning Engine

Scans the memory space used by a running process to expose malicious behavior that could be hiding through code obfuscation.

Runtime
Details
  • Scans process memory for known shellcode patterns
  • Detects reflective DLL injection in memory
  • Catches unpacked payloads after execution begins
🔗

AMSI Integration Engine

Deep in-app integration defeats code obfuscation. Blocks malicious behavior of scripts client-side, defeating fileless and in-memory attacks.

Pre-execution
Details
  • Covers PowerShell, VBScript, JScript, VBA macros
  • Sees code after all deobfuscation layers
  • Client-side blocking before cloud roundtrip

Heuristics Engine

Identifies file characteristics that have similarities with known malicious characteristics to catch new threats or modified variants.

Pre-execution
Details

Dynamically unpacks malware and examines runtime behavior. Scans both behavior during emulation and memory content at the end to defeat polymorphic malware and packers.

Pre-execution
Details

  • Exposes polymorphic payload post-deobfuscation
  • Lightweight — runs in milliseconds

Pairs of client-side and cloud-side models perform advanced analysis of scripting behavior pre- and post-execution. Separate model pairs for each scripting engine.

Cloud ML

Deep Learning

Suspicious files detonated in a cloud sandbox. Deep learning classifiers analyze observed behaviors to block attacks.

Queries domain-expert reputation sources across Microsoft — SmartScreen URL models, Defender for Office 365 email knowledge, and the Microsoft Intelligent Security Graph.

Reputation

💡 Why Hybrid Detection Engines Matter
🛡️

Layered Defense = No Single Point of Failure

4 client engines + 4 cloud engines working in parallel. If the heuristics engine misses it, the emulation engine catches it. If both miss, the cloud detonation sandbox gets it. Attackers have to evade all 8 layers.

⏱️

"Block at First Sight"

Unknown files are held from executing while the cloud ML classifies them — verdict in seconds. This stops zero-day malware before it runs, not after damage is done.

🧬

Defeats Modern Evasion

Polymorphic malware, fileless attacks, obfuscated scripts — traditional signatures can't keep up. The emulation engine unpacks packers, AMSI defeats encoding, and ML models classify behavior patterns, not just file hashes.

🌐

78 Trillion Daily Signals

The cloud engines leverage threat intelligence from Microsoft's entire ecosystem — Windows, Office, Azure, Xbox, Bing. A URL seen in a phishing email in Japan can be blocked on your endpoint in seconds.

🔄 Layer 3

Telemetry Pipeline

How behavioral data flows from endpoint to cloud to detection to response. Compressed, encrypted, streamed over HTTPS (~5–20 MB/day per device).

1

📡 Collect — Kernel & User-Mode Hooks

MsSense.exe collects process, file, network, registry, logon, DLL, memory, and AMSI events via ETW, minifilter drivers, WFP callouts, and kernel callbacks. Linux: eBPF tracepoints.

2

🔒 Transmit — Compressed & Encrypted

Telemetry deduplicated, compressed, and encrypted on-device, then streamed over HTTPS to Azure datacenters. Internal throttling prevents duplicate event flooding.

3

⚡ Process — Cloud Correlation Engine

Backend correlates signals across ALL tenant devices via the Intelligent Security Graph (78T+ daily signals). Cross-product correlation with Identity, Office 365, and Cloud Apps.

4

🚨 Detect — Alert & Incident Generation

ML models and detection rules identify attack patterns. Alerts generated with MITRE ATT&CK mappings. Alerts sharing attack techniques or attacker are aggregated into incidents.

5

🛡️ Respond — Automated or Manual Actions

Response actions pushed back to endpoint via sensor channel: isolate device, quarantine file, kill process, run AV scan, collect investigation package. Works even on isolated devices.

6

💾 Store — Queryable Telemetry

Raw telemetry stored 30 days in Advanced Hunting tables. Alert/incident data retained 180 days. Enables "travel back in time" to investigate the start of an attack.

💡 Why the Telemetry Pipeline Matters
🔍

Cross-Device Visibility

No single endpoint sees the full attack. Lateral movement from Device A → Device B is invisible to each device alone. The cloud pipeline correlates across your entire fleet to detect multi-stage attacks.

⏪

6 Months of Time Travel

When a breach is discovered months later, you can investigate back to the original compromise. Most incidents aren't detected the same day — MDE keeps the data for when you need it.

🔒

Response Works Even on Isolated Devices

When you isolate a compromised device, the MDE sensor channel stays active. You can still collect evidence, run scans, and remediate — without giving the attacker network access.

📋

Compliance & Forensics

Regulatory frameworks (NIST, ISO 27001, HIPAA) require evidence of detection and response capabilities. MDE's 180-day retention provides the audit trail you need for compliance and legal proceedings.

⚖️

💡 Why Automated Investigation Matters
⏱️

Minutes, Not Hours

Mean Time to Respond drops from hours to minutes. AIR does in 60 seconds what a Tier 1 analyst does in 45 minutes — investigate the alert, check other devices, and remediate. That speed gap is the difference between containment and breach.

📉

Reduces Alert Fatigue

SOC teams face thousands of alerts daily. AIR automatically resolves true positives and dismisses false positives — your analysts only see what requires human judgment.

🔀

Scope Expansion Catches What Humans Miss

When AIR finds the same threat on 5 other devices, it adds them to the investigation automatically. Human analysts often focus on the alerted device and miss that the attacker has already spread.

⚙️

Configurable Trust Levels

Not ready for full automation? Start with Semi — AIR investigates and recommends, but waits for your approval before acting. Build confidence before going full auto.
