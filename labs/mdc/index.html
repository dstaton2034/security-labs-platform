🏛️ Foundation

CNAPP — The Three Pillars

Defender for Cloud is a Cloud Native Application Protection Platform — a unified solution combining posture management, development security, and workload protection into a single platform that secures applications across their entire lifecycle.

🛡️
CSPM
Cloud Security Posture Management

Continuously checks and improves the security posture of your cloud resources. Finds misconfigurations, tracks compliance, scores your security posture, and models attack paths.

Prevent Posture
💻
DevSecOps
Development Security Operations

Manages code-level security across multicloud and multi-pipeline environments. Scans IaC templates, detects secrets in code, and provides PR annotations — shifting security left into development.

Shift Left Code
⚔️
CWPP
Cloud Workload Protection Platform

Defends running workloads — VMs, containers, databases, storage, serverless, APIs — from active threats. Provides workload-specific security alerts and response capabilities at runtime.

Detect Respond
💡 Why CNAPP Matters
🔗

One Platform, Full Lifecycle

Instead of separate tools for posture, code scanning, and runtime protection — CNAPP unifies them. A vulnerability found in code can be traced through deployment to the running workload in one console.

⬅️

Shift Left, Not Just Shield Right

Fixing a misconfiguration in an IaC template costs minutes. Fixing it in production after an attacker exploits it costs millions. DevSecOps catches issues before they deploy.

☁️

Multicloud by Default

Most organizations run Azure + AWS + GCP. Defender for Cloud provides a single pane of glass across all three — same recommendations, same policies, same compliance standards.

📊

Posture + Protection Together

CSPM tells you "this storage account is publicly accessible." CWPP tells you "someone is actively downloading data from it." Together, they give you context that neither provides alone.

💡 Why CSPM Matters
🔥

Misconfiguration = #1 Cloud Risk

Most cloud breaches aren't zero-days — they're open storage accounts, overly permissive IAM roles, and unencrypted databases. CSPM finds these before attackers do.

📈

Secure Score for Leadership

CISOs need a number they can report to the board. Secure Score provides a trackable, improvable metric that shows security posture trending over time.

⚖️

Compliance on Autopilot

Instead of manually mapping controls to CIS/NIST/PCI, Defender CSPM continuously assesses compliance and shows exactly which resources fail which control — with remediation steps.

🔍

Agentless = Zero Friction

Agentless scanning takes disk snapshots and analyzes them externally. No agent to deploy, no performance impact on workloads, no maintenance overhead. Security teams can scan everything without asking app owners for permission.

Workload-Specific Beats Generic

🔄

Posture + Runtime = Complete

CSPM says "your storage is misconfigured." CWPP says "someone is actively exploiting that misconfiguration right now." You need both — prevention AND detection.

Pay Only for What You Use

🗺️ Attack Path Analysis

Automatically discovers paths an attacker could take from an entry point (internet-facing resource with a vulnerability) to your crown jewels (sensitive data, critical workloads). Prioritizes paths by risk — not just individual vulnerabilities.

🔎 Cloud Security Explorer

Build custom graph queries: "Show me all VMs with public IPs that have critical vulnerabilities AND access to storage accounts containing sensitive data." Like KQL, but for your cloud architecture.

💡 Why Attack Paths Matter
👁️

See the Attacker's Perspective

Individual findings like "VM has a critical CVE" or "storage allows public access" seem unrelated. Attack path analysis connects them: that CVE is the entry point to your sensitive data. Context changes everything.

🎯

Fix the Right Thing First

You have 500 critical vulnerabilities. Which one matters? The one that's internet-facing, has a public exploit, and leads to your customer database. Attack paths answer "what should I fix NOW."

🕸️

Graph-Based, Not List-Based

Traditional tools give you a flat list of findings. The security graph shows relationships — identity → resource → data → network. You see blast radius, not just individual issues.

🔗

Break One Link, Kill the Path

You don't have to fix every finding in a path. Remove the public IP, or fix the over-privileged identity, or enable storage encryption — breaking any single link eliminates the entire attack path.

🔄 Architecture

How Defender for Cloud Collects Data

Multiple collection methods — agentless, agent-based, and connector-based — feed into the Cloud Security Graph for unified posture and protection.

1

🔌 Cloud Connectors

Azure subscriptions auto-connect natively. AWS uses CloudTrail + SSM connector. GCP uses Security Command Center + service accounts. All configuration is agentless — no agents deployed in the target cloud.

2

🔍 Agentless Scanning

Takes snapshots of VM disks and scans them externally for vulnerabilities, secrets, and malware. Zero performance impact — the workload never knows it's being scanned. Works across Azure, AWS, and GCP.

3

🛡️ Agent-Based (Optional)

For deep workload protection: Azure Monitor Agent or MDE sensor. Provides file integrity monitoring, adaptive application controls, just-in-time VM access, and full EDR capabilities.

4

🌐 Azure Arc (Hybrid)

Projects on-premises and other-cloud machines as Azure resources. Extends Defender for Cloud to anywhere — your datacenter, edge, or other cloud providers. Same policies, same recommendations.

5

💻 DevOps Connectors

Connects to GitHub, Azure DevOps, and GitLab. Scans IaC templates for misconfigurations, detects secrets in code, provides PR annotations so developers fix issues before merge.

6

🕸️ Cloud Security Graph

All data feeds into the unified security graph — resource inventory, vulnerabilities, identities, network paths, sensitive data locations. Powers attack path analysis, security explorer, and risk prioritization.

💡 Why Multicloud Matters
📊

93% of Enterprises Are Multicloud

If your security tool only covers Azure, you're blind to AWS and GCP risks. Attackers don't care which cloud you use — they exploit the weakest link across all of them.

📋

One Policy, Three Clouds

"All storage must be encrypted" — applied consistently across Azure Blob, AWS S3, and GCP Cloud Storage from a single console. No per-cloud tool sprawl.

🔗

Cross-Cloud Attack Paths

An attacker could pivot from a compromised AWS EC2 instance to an Azure storage account via a shared service principal. Only a multicloud tool sees that path.

💰

Consolidation Saves Money

Replace separate CSPM tools per cloud (Prisma, AWS Security Hub, GCP SCC) with one platform. Fewer vendors, fewer consoles, lower total cost.

Defender for Cloud Ecosystem

Integrates with the full Microsoft security stack and third-party tools for unified operations.

Microsoft Sentinel
SIEM integration — all MDC alerts stream into Sentinel as security incidents.
Bi-directionalAuto-enabled
How it integrates →
  • Data connector: Enable the "Microsoft Defender for Cloud" connector in Sentinel — one click, all alerts flow in
  • Bi-directional sync: Alert status changes in Sentinel sync back to MDC and vice versa
  • Continuous export: Stream recommendations, secure score changes, and regulatory compliance data to Log Analytics
  • Workbooks: Pre-built Sentinel workbooks visualize MDC data — posture trends, alert heatmaps, compliance dashboards
  • Automation: Sentinel playbooks (Logic Apps) trigger on MDC alerts — auto-remediate, notify teams, create tickets
  • Hunting: Query MDC security data alongside endpoint, identity, and email data in one KQL workspace
🛡️
Defender XDR
MDC alerts appear in the unified Defender XDR incident queue alongside endpoint and identity alerts.
Unified PortalAuto-correlation
How it integrates →
  • Unified incidents: Cloud alerts are correlated with endpoint/identity alerts into single incidents in security.microsoft.com
  • Auto-correlation: A compromised user identity (Defender for Identity) + compromised VM (MDC) = one incident, not two
  • Shared investigation: Investigate cloud resource alerts alongside device timelines and user activity in one view
  • Attack disruption: XDR can automatically contain threats spanning cloud + endpoint + identity
  • No config needed: MDC alerts flow into XDR automatically when both are licensed
🖥️
Defender for Endpoint
Defender for Servers auto-deploys MDE on every server — full EDR with one toggle.
EmbeddedAuto-deploy
How it integrates →
  • Auto-provisioning: Enable Defender for Servers → MDE sensor auto-installs on all Azure VMs and Arc-connected servers
  • Unified vulnerability assessment: MDE's TVM feeds vulnerability data back into MDC recommendations and attack paths
  • Shared alerts: MDE endpoint alerts appear in both the MDC alerts page and the XDR incident queue
  • License included: MDE P2 license is included with Defender for Servers P2 — no separate purchase
  • Cross-platform: Works on Windows Server, Linux (Azure + Arc), and even AWS EC2/GCP Compute via Arc

Azure Policy

  • Custom policies: Create custom Azure Policies that feed into MDC as custom recommendations with secure score impact
  • Regulatory compliance: Compliance standards (CIS, NIST, PCI-DSS) are implemented as Policy initiatives that MDC tracks
